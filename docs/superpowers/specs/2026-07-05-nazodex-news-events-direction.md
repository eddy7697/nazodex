# NazoDex 新聞與事件功能——方向討論記錄

> 2026-07-05 與 Vincent 的討論整理。性質是**方向文件**（direction doc），非實作規格；
> 各 milestone 動工前仍走既有 brainstorming → spec → plan 流程。

## 背景與動機

Vincent（投資新手）的原始提問：能否爬取國內外產業/財經新聞、產業領袖與國際領袖發言，
收攏後讓策略推薦的「預測明朗度」更好？

討論後把需求重新定位為：**不是更準的預測，而是一個幫使用者把雜訊分類、
判斷「這則消息該不該當回事」的引導系統。** 核心痛點：

- 新手身邊充斥散戶圈的「事前/事後諸葛」，假訊號到處都是，市場沒有教科書也沒有目錄。
- 個人不可能全天候監控市場；能做的是跟隨資訊優勢者（法人）的**足跡**，而非說法。
- 需要的是**更有效率整理熱點新聞**的功能，以及接軌市場知識的引導。

## 為什麼「爬新聞餵模型做預測」是死路

1. **新聞大多已反映在價格裡**：免費源 + 每日批次的架構，抓到的是「解釋昨天為什麼漲」的後見之明，不是預測訊號。
2. **更多資訊 ≠ 更明朗**：財經新聞雜訊比例極高，餵進模型等於加一個看起來厲害、實際加雜訊的因子。
3. **無法驗證（最致命）**：現有五因子皆為結構化數字，歷史累積後可回測；新聞情緒因子需要歷史新聞存檔＋當時情緒標註，拿不到。放無法回測的因子進評分會侵蝕整個分數的可信度。
4. **國際領袖發言是總經層級訊號**：影響大盤方向與風險偏好，與策略頁的截面選股是不同維度。

**結論定位**：新聞/事件是「**呈現給人做判斷**」的脈絡資訊，不進評分引擎。
劃線原則：**進引擎的東西要能被回測；給人看的東西只要誠實標註來源。**
這與 NazoDex 設計哲學（使用門檻低、操作上限高、深度資訊漸進式揭露）一致。

## 知識框架（未來要內建進產品的教材）

### 波動成因四層模型

| 層級 | 例子 | 影響持續性 |
|------|------|-----------|
| 總經／資金面 | 升降息、匯率、地緣政治 | 數月～數年，影響所有股票 |
| 產業面 | 景氣循環、供需反轉、技術典範轉移 | 數月～數季 |
| 個股基本面 | 財報、接單、擴廠、經營權變動 | 數季，真正決定長期股價 |
| 籌碼／情緒面 | 法人進出、新聞熱度、當沖、名嘴喊單 | 數天～數週，大多退潮 |

**長遠性判準**：這件事有沒有改變公司未來賺錢的能力？有 → 影響長；
只是改變「感覺」→ 幾天內被消化。散戶圈的話術九成發生在第四層。

### 假訊號過濾器（可用性 heuristics）

- 新聞見報、股價沒反應 → 市場已判定不重要或早已反映。
- 利多見報、股價反跌 → 常是「利多出盡」，先知者正倒貨給看新聞進場的人。
- 只有故事、沒有數字（「聽說要接大單」）→ 第四層情緒燃料。
- 事後諸葛萬用句式（「就說會漲吧」）→ 社交表演，非資訊。

### 法人足跡原則

法人不是「操縱」而是「先讀完考卷才作答」。跟隨其**足跡**（買賣超數字，錢不說謊）
比跟隨其**說法**（受訪發言）可靠。既有籌碼因子本質上已是
「資訊優勢者讀完所有新聞後用錢投票」的代理指標——免費、結構化、可回測。

## Milestone 路線（不對稱拆分：只把最近一步拆到 task 級）

> 原則：M3 的正確做法會被 M1/M2 上線後的實際體驗改寫（雜訊量、標籤價值），
> 現在拆細是在規劃一定會過期的細節。每個 milestone 獨立可交付，
> 各對應一個 feature branch（branch → 測試 → PR 流程）。

### M1：官方事件層（第一步，最小最實）

個股頁顯示 MOPS 重大訊息 + 注意/處置股標記。零 NLP、官方免費源。
價值：自選股突然大跌時，第一眼先看官方公告，過濾掉一半恐慌。

Task 拆分：

1. **資料源探勘**（唯一有不確定性的 task，先做）：確認 MOPS 重大訊息取得方式
   （OpenAPI `t187ap04_L` 系列）與注意/處置股名單端點；驗證欄位、更新頻率、限流。
2. `lib/announcements/` 抽象層：`getAnnouncements(symbol)` + 快取
   （比照 market-overview 的 10min 快取與單源容錯模式），TDD。
3. 注意/處置股旗標：`getAlertFlags(symbols)`，整包快照 + 記憶體 join（比照 screener 模式）。
4. API route + 個股頁 UI：`app/stock/[symbol]` 加「近期公告」區塊，
   手機卡片/桌面表格響應式，空狀態沿用 `EmptyState`。
5. 自選股清單上的處置股警示徽章（警訊不用點進個股頁就看得到）。
6. prod 驗證（middleware matcher 雷：新增 public 素材要同步排除清單）。

### M2：個股新聞牆（粗顆粒）

財經媒體 RSS（如鉅亨網）聚合、依代號歸戶，純列表、標時間、附連結。
不解讀、不打分，先解決「不用自己去 Google」。依賴 M1 的個股頁事件區版位。

### M3：知識標籤（粗顆粒，本路線的核心價值）

用 LLM 為每則新聞貼標籤——不預測漲跌，而是標：

- 屬於四層模型的哪一層（總經/產業/基本面/籌碼題材）
- 「事實型公告」vs「媒體解讀」

等於把四層模型內建到新聞牆，使用者每天看盤同時被動學會分辨雜訊——
**產品本身成為那本不存在的教科書。** 依賴 M2 累積的真實新聞流來調校。

### M4：新聞 × 籌碼對照（粗顆粒）

新聞旁顯示當日法人買賣超，養成「聽到消息先看聰明錢有沒有跟」的反射。
依賴 M2 + 既有策略頁 T86 資料。

## YAGNI / 明確不做

- 新聞情緒因子進策略評分引擎（無法回測，侵蝕可信度）。
- 爬取領袖發言做量化訊號（總經層級、話術雜訊高）。
- 自建爬蟲對抗反爬（優先官方 OpenAPI 與 RSS）。
- 即時新聞推播（v1 全部是拉式呈現）。

## 開放問題（動工時再決）

- RSS 源選擇與新聞→代號歸戶的準確率（M2）。
- LLM 標籤的成本/延遲與跑批時機（M3；可能併入每日 ingest CronJob）。
- 重大訊息是否需要落 DB 表（歷史查閱）還是純快取即可（M1 task 1 探勘後決定）。
